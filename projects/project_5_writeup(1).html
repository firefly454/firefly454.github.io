Udacity Project 5: Identify Fraud from Enron Email¶

Erica J. Kim

March 2016

Project Summary and Goals¶

The Enron scandal, which was revealed in 2001 and led to the bankruptcy of the Enron Corporation in 2002, remains one of the largest cases of corporate fraud in American history. In the ensuing Federal investigation, thousands of emails, detailed financial data, and other normally confidential information became part of the public record. Among the hundreds of employees at Enron and related companies, a few handful were either:

  • indicted
  • reached a settlement or entered a plea agreement with the government
  • testified in exchange for prosecution immunity.

The purpose of this project is to classify actual Enron employees as being or not being "persons of interest" ("POIs"), i.e. among the few that did one of the above actions, using data made public from the investigation.

Data Exploration & Outlier Investigation¶

The primary dataset is a dictionary containing financial and email information for 143 employees, of which 18 are POIs. Example financial features are salary and exercised stock options, with 14 financial features in total. While some financial features were densely populated (e.g. salary, total payments, and total stock value), other financial features had had many "-" values (e.g. loan advances, director fees, and restricted stock deferred). I interpreted these "-" values as intended to be $0.00, and not missing values (the original spreadsheet from which the data came from was used in the actual Enron case (Exhibit 3b.2, FindLaw.com) and presumably completed), and transformed them that way. Thus, in that sense, there were no missing values among the financial featuers. Example email features are the number of emails sent and the number of emails received, with 5 email features in total, and was engineered by Udacity. The secondary and optional dataset for this project is the entire public Enron email corpus, consisting of actual emails (both content and metadata) to and from Enron employees. Using machine learning - in particular, a supervised classification algorithm - we can use these datasets to potentially predict POIs, either those who we already know are POIs, or those who have yet to be discovered...

Feature Engineering¶

The original dataset features were:

  • salary
  • bonus
  • long_term_incentive
  • deferred_income
  • deferral_payments
  • loan_advances
  • other
  • expenses
  • director_fees
  • total_payments
  • restricted_stock_deferred
  • exercised_stock_options
  • restricted_stock
  • total_stock_value
  • to_messages
  • from_messages
  • from_poi_to_this_person
  • from_this_person_to_poi
  • shared_receipt_with_poi

"total_payments" is the sum of the previous 9 pay/salary-related features, while "total_stock_value" is the sum of the previous 3 stock-related features. All of the financial features have absolute dollar (USD) units, while all of the email-related features have # of email units. Thus, it seemed natural to engineer new features that reflected specific ratios, rather than absolute dollar values or # of emails. The new features I created were:

  • from_poi_ratio (= from_poi_to_this_person/to_messages)
  • to_poi_ratio (= from_this_person_to_poi/from_messages)
  • shared_poi_ratio (= shared_receipt_with_poi/to_messages)
  • salary_ratio (= salary/total_payments)
  • bonus_ratio (= bonus/total_payments)
  • deferred_income_ratio (= deferred_income/total_payments)
  • longterm_ratio (= long_term_incentive/total_payments)
  • exercised_stock_ratio (= exercised_stock/total_stock_value)
  • restricted_stock_ratio (= restricted_stock/total_stock_value)
  • stock_fraction (= total_stock_value/(total_stock_value + total_payments)

Ernst & Young email keyword analysis¶

Inspired by this article from ComputerWeekly.com (http://www.computerweekly.com/news/2240175638/Ernst-Young-email-keyword-analysis-identifies-ID-fraudsters) as well as a few threads in the Udacity forum, I additionally created another feature, "flagged_word_counter", for each person in the dataset. "Flagged_word_counter" is an integer variable containing the number of times a 'bad' phrase is used in a person's outgoing/sent email. The list of bad phrases (26 total) was taken directly from the aforementioned article; some examples are:

  • "cover up"
  • "write off"
  • "failed investment"
  • "off the books"
  • "nobody will find out"
  • "grey area"
  • "special fees"
  • "they owe it to me"
  • "not hurting anyone"

After engineering new features, the total feature list contained 29 features (not including "poi" label).

Feature Selection¶

While there are many different algorithms for choosing features, such as Principal Component Analysis or KBestPercentile, I decided to implement sklearn's SelectKBest method, which takes in the list of features as well as an integer k, and then outputs the k highest scoring features via univariate statistical tests (ANOVA F-values in this case). However, what should we choose as k? Because I had no initial idea how large or small k should be, I first ran SelectKBest with k=29 in order to obtain the scores for all of the features, and then re-ordered the feature list by score. I coded a for-loop, ranging from k=1 to k=29. In each loop iteration, I created a temporary feature list with ranked features 1 to k, input this new feature list into the tester code in my program (using AdaBoost classifier with default parameter values, and StratifiedShuffleSplit cross-validation), and output recall and precision scores. I could then see how precision and recall changed with each value of k, and even plot these numbers, as below:

Testing different classifiers¶

Several different classification algorithms were tested on the dataset, using the k-best selected features list. I decided to forego feature scaling since most of the classifiers (all but K-Nearest Neighbors) I was testing out did not actually benefit from feature scaling. I tested primarily ensemble methods (Gradient Boosting, AdaBoost, and Random Forests), as well as K-Nearest Neighbors and Support Vector Machines. In terms of classifier parameters, I did some basic parameter tuning as well, since classifier performance can depend greatly on parameter values. A variable for the highest-scoring classifier was continually updated automatically within the loop as well.

Recall scores are shown for each classifier and the particular parameter values used in each case, below (RBF and linear support vector machine clasifiers resulted in zero true positive predictions and thus no scores). Based on these results, I chose to use the AdaBoostClassifier. Some things to note: I used 300 estimators for GradientBoosting and AdaBoost, and only 20 estimators for RandomForest - this is simply because the computation time in training the RandomForest classifier was much higher than those for the other ensemble methods. Additionally, since we performed feature selection using AdaBoostClassifier, this algorithm obviously potentially had an advantage over the other algorithms.

Tuning classifier¶

AdaBoostClassifier worked quite well without having to be extensively tuned, as shown above. However, we can of course potentially achieve even better scores with some additional tuning of parameters. Without tuning, we would just be using AdaBoost's default parameter values. With tuning, however, we could potentially create a better performing model, i.e. a model that fits the general data better and thus has better predictive power.

AdaBoost's primary parameters are: the underlying base estimator (default is DecisionTreeClassifier), the maximum number of estimators at which boosting is terminated (default value is 50), and the shrinkage rate that each estimator is limited by (default value is 1). We can additionally tune the base estimator - the main parameters to tune for Decision Trees are the max_depth and the min_samples_leaf, both which control how complex the tree gets. These parameters, in general, will determine how underfitted or overfitted the model is. I again used sklearn's GridSearchCV, but expanded the 'parameters' dictionary to search through more parameter values: learning rate, base_estiamtor's max_depth and base_estimator's min_samples_leaf. I also continued to use the same cross-validation method and scoring metric as before. This additional tuning yielded optimal parameter values of:

  • 'learning_rate': 1
  • 'base_estimator__max_depth': None
  • 'base_estimator__min_samples_leaf': 1

Algorithm Performance¶

Final testing of our algorithm on the data (again, using sklearn's StratifiedShuffleSplit method for cross validation), resulted in the following scores:

  • precision: 0.34
  • recall: 0.34

Such scores are quite impressive, considering the very small dataset and the limited # of original features. The tools used in this data science project could also be generalized to a wide variety of other classification problems. In particular, any sort of fraud detection problem would utilize similar steps and tools, and would be similar to this problem in the sense that it is also a binary classification problem (something is either fraudulent or not), and the classes are generally very imbalanced (# of fraud cases is most likely much less than the # of real cases).
